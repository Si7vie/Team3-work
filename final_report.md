# 股权激励计划字段抽取 — 阶段性报告

**项目方向**：基于巨潮资讯股票期权激励计划草案，通过文本智能抽取构建股权激励慷慨度评价体系\
**数据来源**：巨潮资讯网（CNINFO），2021年1月至2025年12月，半导体/电子/计算机/医药/高端制造行业\
**数据补充**：通过东方财富API，获取公司总人数用于计算股权激励人数比例\
**样本规模**：200 份股权激励计划草案 PDF 公告\
**当前阶段**：规则基线（Rule-based Baseline）字段抽取已完成

***

## 一、PDF 转 Markdown：MinerU 三阶段批量解析

### 1.1 为什么需要 PDF → Markdown？

PDF 是给人阅读的排版格式（含复杂表格、分栏、嵌入图片），计算机无法直接处理。Markdown 是纯文本格式，保留标题/段落结构，可直接被正则表达式和 LLM 消费。因此，**PDF → Markdown 是所有下游任务的先决条件**。

我们选用 **MinerU**（OpenXLab 云端文档解析服务），其核心能力：输入 PDF 网址 → 输出格式化 Markdown 文件。底层采用 VLM（视觉语言模型），能够理解文档的视觉布局。

### 1.2 三阶段流水线架构

整个转换过程由**三个独立阶段接力完成**，而非单次 API 调用：

```
╔══════════════════════════════════════════════════════════════════╗
║  阶段一：批量提交（scripts/batch_parse.py）                       ║
║  ────────────────────────────────────────                       ║
║  输入: data/metadata/metadata.csv（200条元数据，含pdf_url列）     ║
║  动作: 遍历每行 → POST pdf_url 至 MinerU API                     ║
║  API:  https://mineru.net/api/v4/extract/task/batch             ║
║  参数:  model_version="vlm", language="ch"                      ║
║  产出: batch_id（每个PDF对应一个任务编号）                         ║
║  输出: data/batch/batch_mapping.csv（doc_id ↔ batch_id 映射）   ║
╚══════════════════════════════════════════╦═══════════════════════╝
                                           ↓
╔══════════════════════════════════════════════════════════════════╗
║  阶段二：轮询查询（scripts/query_batch.py）                       ║
║  ────────────────────────────────────────                       ║
║  输入: data/batch/batch_mapping.csv                             ║
║  动作: 遍历每个 batch_id → GET MinerU 结果查询接口                 ║
║  API:  https://mineru.net/api/v4/extract-results/batch/{id}     ║
║  检查: state 字段 → "done"（完成）/ "processing"（处理中）        ║
║  产出: full_zip_url（解析结果 ZIP 包的下载链接）                   ║
║  输出: data/batch/result_mapping.csv（doc_id+state+zip_url）    ║
╚══════════════════════════════════════════╦═══════════════════════╝
                                           ↓
╔══════════════════════════════════════════════════════════════════╗
║  阶段三：下载解压（scripts/download_md.py）                       ║
║  ────────────────────────────────────────                       ║
║  输入: data/batch/result_mapping.csv（仅 state=="done" 的行）    ║
║  动作: GET full_zip_url → 下载 ZIP → 解压 → 提取 markdown.md     ║
║  存放: markdown/md/<doc_id>.md                                  ║
║  输出: 200个 .md 文件（每个对应一份公告全文）                       ║
╚══════════════════════════════════════════════════════════════════╝
```

### 1.3 核心代码示例

**阶段一（提交任务）**：
```python
# scripts/batch_parse.py
for _, row in df.iterrows():
    response = requests.post(
        "https://mineru.net/api/v4/extract/task/batch",
        headers={"Authorization": f"Bearer {os.getenv('MINERU_API_KEY')}"},
        json={
            "files": [{"url": row["pdf_url"], "data_id": str(row["doc_id"])}],
            "model_version": "vlm"
        }
    )
    batch_id = response.json()["data"]["batch_id"]
    # 存入 batch_mapping.csv
```

**阶段二（轮询状态）**：
```python
# scripts/query_batch.py
url = f"https://mineru.net/api/v4/extract-results/batch/{batch_id}"
result = requests.get(url, headers=headers).json()
state = result["data"]["extract_result"][0]["state"]  # done / processing
full_zip_url = result["data"]["extract_result"][0].get("full_zip_url", "")
```

**阶段三（下载解压）**：
```python
# scripts/download_md.py
r = requests.get(zip_url)
with open(f"downloads/{doc_id}.zip", "wb") as f:
    f.write(r.content)
with zipfile.ZipFile(zip_path, "r") as zip_ref:
    zip_ref.extractall(f"downloads/{doc_id}")
# 将 downloads/{doc_id}/markdown.md → markdown/md/{doc_id}.md
```

### 1.4 输出

200 份 Markdown 文件存放于 `markdown/md/` 目录，文件名为 `<doc_id>.md`（如 `1224831097.md`），其中 `doc_id` 为巨潮资讯公告唯一标识符。

### 1.5 遇到的主要问题

- **异步等待**：MinerU 不即时返回结果，需轮询 `state == "done"`（单份平均等待 30-90 秒）
- **批量限制**：单次提交有文件数量上限，200 份需分批（每批约 20 份）
- **ZIP 解压**：返回的是 ZIP 包而非单个 .md 文件，内含 markdown.md + 图片文件夹，需解压后整理
- **部分失败处理**：2 份公告（`1219198504` 东软集团、`1219389670` 中岩大地）解析后仅为激励对象名单附录，非完整草案正文

## 二、Pydantic Schema 设计

为保障字段类型安全和后续校验，使用 Pydantic v2 定义了结构化数据模型。

### 2.1 Schema 定义 (`src/schemas.py`)

```python
class EquityIncentiveExtract(BaseModel):
    doc_id: str                              # 公告唯一ID
    company_name: str                        # 公司名称
    stock_code: Optional[str] = None         # 股票代码
    grant_amount: float                      # 拟授予总数量（万份）
    grant_ratio: Optional[float] = None      # 占总股本比例（%）
    participant_count: int                   # 首次授予总人数
    exercise_price: float                    # 行权价格（元/份）
    market_price: Optional[float] = None     # 公告前1日股票交易均价（元）
    discount_rate: Optional[float] = None    # 定价折扣率（%）
    waiting_period: Optional[float] = None   # 等待期（月）
    validity_period: float                   # 激励计划最长有效期（月）
    evidences: list[Evidence]                # 提取证据文本
```

### 2.2 设计考量

- **必填字段**（`grant_amount`、`participant_count`、`exercise_price`、`validity_period`）对应激励计划草案中的法定披露项，每份公告必须包含
- **可选字段**（`market_price`、`discount_rate`、`waiting_period`）在某些公告中可能缺失（如纯自主定价不引用市场均价）
- `evidences` 字段保存原始文本片段（前 500 字符），供人工核验追溯

### 3.3 提取方法概述

采用纯正则表达式（Regex）方案，每个字段编写多条正则规则，按优先级从高到低依次匹配：

| 字段 | 正则规则数 | 核心思路 | 提取策略 |
|------|:---:|------|------|
| `grant_amount` | ~15 | 匹配"万份"/"万股"时保留原值，匹配"份"/"股"时除以 10000 | 多模式匹配 + 单位判断 |
| `grant_ratio` | ~10 | 优先在"授予"相关上下文 1000 字符内匹配百分比 | 上下文限定 + 合理性过滤(≤50%) |
| `participant_count` | ~15 | 匹配"共计 X 人"、"X 名激励对象"等模式 | 千分位处理 + 范围验证(1~100000) |
| `exercise_price` | ~8 | 匹配"行权价格为 X 元"等表述 | 多关键词覆盖 |
| `market_price` | ~20 | 三层优先级：直接价格 → 反推 → 直接声明 | ★最复杂：严格限定"前1个交易日"上下文 |
| `discount_rate` | 计算得出 | `(1 - exercise_price / market_price) × 100` | 依赖 market_price 准确性 |
| `waiting_period` | ~10 | 匹配"等待期为 X 个月/分别为 X 个月"等 | 多阶段取首个 + 范围验证 |
| `validity_period` | ~8 | 匹配"有效期最长 X 个月"等表述 | 多表述覆盖 |

### 3.4 LLM 备用通道

当正则无法覆盖边缘 case 时（如纯自主定价不引用市场均价），可切换至 LLM 模式：

```bash
python src/extract_fields.py --config configs/workflow.yaml --method llm
```

LLM 将公告文本 + 字段说明发送至 Qwen3-8B（通过 SiliconFlow API），由模型直接输出结构化 JSON。目前作为正则的 fallback 兜底方案，覆盖正则遗漏的少量样本。

## 四、编写 Baseline 中遇到的坑

### 4.1 `grant_amount`：单位混淆

股权激励公告中，授予数量至少有四种表述方式：

| 表述     | 示例                              | 实际含义          | 应转换为       |
| ------ | ------------------------------- | ------------- | ---------- |
| **万份** | "拟授予股票期权 **3,186.2261 万份**"     | 3186.2261 万份  | 直接使用 ✅     |
| **万股** | "授予权益总计 **1,200.00 万股**"        | 1200 万份       | 直接使用 ✅     |
| **份**  | "可授予的 A 股期权上限为 **5,726,100 份**" | 572.61 万份     | ÷ 10000 ⚠️ |
| **股**  | "拟向激励对象授予 **116,407,025 股**"    | 11640.7025 万份 | ÷ 10000 ⚠️ |

**教训**：正则匹配时必须检查匹配到的完整文本（`m.group(0)`）中是否包含"万"字。如果没有"万份"或"万股"，说明单位是"份"或"股"，必须除以 10000。早期的正则规则遗漏了这一判断，导致 `grant_amount` 被提取为 5,726,100（万份）而非正确的 572.61（万份）。

### 4.2 `participant_count`：千分位逗号

部分公告中激励人数使用了带逗号的千分位格式：

> "激励对象共计 **6,302 人**"

早期正则表达式 `([0-9]+)\s*人` 只捕获了逗号前的 `6`，遗漏了 `302`。

**修复**：将正则中的数字匹配从 `[0-9]+` 扩展为 `[0-9,，]+`，并在取值时使用 `.replace(",", "").replace("，", "")` 去除千分位分隔符。此外还需要验证提取的数值在合理范围内（1 < N < 100,000），避免错误匹配到其他数字。

### 4.3 `discount_rate`：依赖 `market_price` 的准确性（最大的坑）

`discount_rate` 不直接从文本中提取，而是由公式计算：

```
discount_rate = (1 - exercise_price / market_price) × 100
```

这意味着 `market_price` 提取错误会直接导致 `discount_rate` 异常。`market_price` 的提取是整个 Baseline 中最复杂的部分，面临以下几类情况：

#### 情况 A：市场价直接给出（最简单）

> "本激励计划草案公告前 1 个交易日公司股票交易均价，**为每股 9.03 元**"

正则：`为每股([0-9.]+)\s*元`

#### 情况 B：市场价被百分比打折后给出（需反推）⚠️

> "公司股票交易均价 **的 75%**，**为每股 7.47 元**"

此时 7.47 是折后价，真正的市场价 = 7.47 ÷ 0.75 = **9.96 元**。

早期代码直接匹配到了 7.47 作为市场价，导致 `discount_rate = (1 - 8.22/7.47) × 100 = -10.04%`（错误）。正确应为 `discount_rate = (1 - 8.22/9.96) × 100 = 17.47%`。

#### 情况 C：市场价先给出，后面跟折后价（容易误判）

> "公司股票交易均价 **每股 90.49 元**的 **50%**，为每股 45.25 元"

此时有两个价格：90.49（市场价）和 45.25（折后价）。正则必须识别出"每股 X 元 的 Y%"的结构——X 才是真正的市场价，而非后面的折后价。

#### 情况 D：区分"前 1 个交易日"和"前 20/60/120 个交易日"

行权价格通常取以下价格中的较高者：

- 前 1 个交易日交易均价
- 前 20/60/120 个交易日交易均价

早期正则没有严格限定"前 1 个交易日"，可能误取到前 20/60/120 日的价格，导致 `market_price` 不准确。

**最终方案**：三层优先级匹配策略

1. **优先级 1**：识别 `(每股X元|X元/股) 的 Y%` 模式 → X 即为市场价
2. **优先级 2**：识别 `交易均价 的 Y%，为/即 Z元` → 市场价 = Z ÷ (Y/100)
3. **优先级 3**：直接匹配 `为每股 X 元`、`即人民币 X 元`、`为 X 元/股` 等

同时，所有匹配都在严格限定"前1个交易日"上下文的文本片段中进行，排除前 20/60/120 日的干扰。

#### 修复效果

修复前的异常 discount\_rate（≥90% 或 ≤-90%）有 7 条，修复后降至 0 条。以盾安环境（1224687306）为例：

- **修复前**：market\_price=7.05, discount\_rate=**-100%**（把限制性股票 50% 折后价当成了市场价）
- **修复后**：market\_price=14.10, discount\_rate=**0%**（正确识别到 14.10 元才是真正的交易均价）

### 4.4 `waiting_period`：多阶段等待期

部分公告的等待期分多个阶段：

> "等待期**分别为**自授予登记完成之日起 **12 个月、24 个月、36 个月**"

早期正则只匹配单个"X 个月"，无法处理"分别为"引导的列表。

**修复**：增加 `等待期分别为[^。；\n]{0,100}?([0-9]+)\s*个?月[、，和及]` 模式，取第一个数值作为最短等待期。

### 4.5 公告类型差异：部分文档仅有附录

200 份 Markdown 文件中，有 2 份（`1219198504` 东软集团、`1219389670` 中岩大地）不是完整的激励计划草案正文，而是**激励对象名单附录**——仅包含人员分配表格，缺少行权价格、有效期等主体内容。这类文档无法用正则提取核心字段，属于数据源本身的结构性缺失。

## 五、Pydantic 校验与错误处理

### 5.1 校验流程 (`validate_results.py`)

```python
for row in rows:
    try:
        model = EquityIncentiveExtract.model_validate(row)
        valid.append(model.model_dump(mode="json"))
    except Exception as e:
        errors.append({"doc_id": row.get("doc_id"), "error": str(e), "raw": row})
```

Pydantic 自动校验：

- `grant_amount` 和 `exercise_price` 是否为有效 float（非 None）
- `participant_count` 是否为有效 int（非 None）
- `validity_period` 是否为有效 float（非 None）
- 可选字段为 None 时不报错

### 5.2 错误输出

校验失败的记录写入 `outputs/logs/validation_errors.jsonl`，每条包含：

- `doc_id`：出错的公告 ID
- `error`：Pydantic 的具体验证错误信息
- `raw`：原始提取结果（含 `evidences` 字段的前 500 字符）

这允许人工检查 `evidence` 文本，判断提取失败是因为正则未覆盖该表述模式，还是因为源文档确实缺少该信息。

## 九、指标方向调整与归一化（Step 3 & 4）

### 9.1 指标方向调整

依据 `evaluation_framework.md`，将所有指标统一为"**越大越慷慨**"的正向指标：

| 指标                  | 原方向    | 调整方式                          | 经济含义                |
| ------------------- | ------ | ----------------------------- | ------------------- |
| `grant_ratio`       | 正向     | 直接使用                          | 授予比例越高，股权激励力度越大     |
| `participant_ratio` | 正向     | 直接使用                          | 参与比例越高，更多员工分享股权激励   |
| `discount_rate`     | 正向     | 直接使用                          | 折扣率越高，员工获得的价格优惠越大   |
| `waiting_period`    | **负向** | 最大值减原值 → `waiting_period_adj` | 等待期越短，员工取得股权激励的速度越快 |
| `validity_period`   | 正向     | 直接使用                          | 有效期越长，长期激励价值越强      |

### 9.2 归一化方法

采用 **Min-Max Scaling** 将各维度映射到 \[0, 1]：

$$x\_{norm} = \frac{x - x\_{min}}{x\_{max} - x\_{min}}$$

### 9.3 归一化后统计特征（完整数据）

下表为 Min-Max 归一化后 5 维度的详细统计特征：

| 指标                   | 均值     | 中位数    | 标准差    | 最小值    | 最大值    |
| -------------------- | ------ | ------ | ------ | ------ | ------ |
| `grant_ratio`        | 0.1707 | 0.0469 | 0.2056 | 0.0000 | 1.0000 |
| `participant_ratio`  | 0.1365 | 0.0469 | 0.2151 | 0.0000 | 1.0000 |
| `discount_rate`      | 0.6214 | 0.5543 | 0.1304 | 0.0000 | 1.0000 |
| `waiting_period_adj` | 0.9301 | 1.0000 | 0.1773 | 0.0000 | 1.0000 |
| `validity_period`    | 0.3673 | 0.3333 | 0.1223 | 0.0000 | 1.0000 |

**指标观察**：

- `grant_ratio` 与 `participant_ratio` 中位数均仅为 **0.0469**，说明超过 50% 的企业在这两个维度上的表现接近底部区间（存在大量的"低规模/低覆盖"企业）；
- `waiting_period_adj` 中位数为 **1.0000**（满值），因为超过 75% 的企业等待期 = 12 个月（最小值），经"最大值减原值"正向化后恰好等于归一化最大值；
- `discount_rate` 均值 0.6214、中位数 0.5543，呈现中间集中的分布特征（大部分企业折扣率在 0%–20% 区间，映射到 0.5 左右）；
- `validity_period` 均值 0.3673，呈中等偏低的均匀分布特征（行业有效期多集中在 48–60 月，但从 12–120 月跨度很大）。

4. 指标间的低相关性是熵权法能够有效区分维度贡献的前提条件，也验证了多维评价体系的合理性（不存在严重的共线性冗余）。

## 十、熵权法赋权（Step 5）

### 10.1 熵权法原理

熵权法是一种**客观赋权法**：指标的信息熵越小，说明该指标在样本间的变异程度越大、提供的信息越多，权重就应越高。

### 10.2 熵权法计算结果

| 指标                   |   信息熵  |     权重     |  排名 |
| -------------------- | :----: | :--------: | :-: |
| `participant_ratio`  | 0.1604 | **52.75%** |  1  |
| `grant_ratio`        | 0.1238 | **40.70%** |  2  |
| `validity_period`    | 0.0107 |    3.52%   |  3  |
| `waiting_period_adj` | 0.0047 |    1.55%   |  4  |
| `discount_rate`      | 0.0045 |    1.48%   |  5  |

### 10.3 权重解读

1. **`participant_ratio`（52.75%）和** **`grant_ratio`（40.70%）合计占 93.45%**：覆盖广度与激励规模是区分企业慷慨度的**两个核心维度**。这与"覆盖率越高、授予比例越大 = 慷慨度越高"的直觉一致。
2. **`validity_period`、`waiting_period`、`discount_rate`** **三者合计仅 6.55%**：并非说明时间与定价不重要，而是说明在当前样本中这三个指标的**变异程度较低**——大多数企业的有效期集中在 48–60 个月，等待期集中在 12 个月，折扣率集中在 0–20%，导致它们区分度弱于规模与覆盖两个维度。
3. **对比人工等权重方案（各 20%）**：熵权法给出的权重方案（Scale+Coverage 合计 \~93%）与人工方案（各维度 20%）在**相对重要性排序**上一致，但在**集中度**上更高——表明数据本身支持"规模和覆盖是主导性维度"的结论。

## 十二、稳健性检验与聚类分析（Step 7）

### 12.1 熵权法 vs 等权重对比

为检验评分体系对权重方案的敏感性，以"人工等权重（5 维度各 20%）"重新计算一次总分，并与熵权法结果进行 Spearman 秩相关检验：

| 对比方法       | Spearman 相关系数 | P 值 |  稳定性  |
| ---------- | :-----------: | :-: | :---: |
| 熵权法 vs 等权重 |   **0.777**   | 0.0 | **高** |

**结论**：两种权重方案的排名相关系数达到 0.777（p < 0.001），说明虽然绝对得分不同，但**企业的相对排名基本稳定**——评分体系对权重方案不敏感，结论具备稳健性。

### 12.2 聚类分析（K-Means）

为进一步探索样本内在结构，对 192 家企业的 5 维度指标进行 K-Means 聚类（K=3），得到三类企业：

|       聚类      | 样本数 | 特征描述                                                       |  平均得分 |
| :-----------: | :-: | ---------------------------------------------------------- | :---: |
| **Cluster 0** |  17 | 高 `participant_ratio`、高 `discount_rate`，但 `grant_ratio` 中等 | 0.150 |
| **Cluster 1** |  56 | `grant_ratio` 最高，但 `participant_ratio` 偏低                  | 0.201 |
| **Cluster 2** | 119 | 各项指标均接近行业中位数，典型企业                                          | 0.171 |

**三类解读**：

- **Cluster 1（56 家，"规模型慷慨"）**：以 `grant_ratio` 突出为特征，主要通过扩大授予规模实现高慷慨度，代表企业如晶澳科技、领益智造。
- **Cluster 0（17 家，"覆盖型慷慨"）**：以 `participant_ratio` 突出为特征，更多采用"全员持股"式的广覆盖方案，代表企业如部分制造业龙头。
- **Cluster 2（119 家，"常规型"）**：行业主流方案，各项指标均接近中位水平。

### 12.3 稳健性小结

1. **权重敏感性低**：熵权法 vs 等权重 Spearman 相关 0.777，结论稳健。
2. **聚类结构清晰**：样本可分为"规模型慷慨""覆盖型慷慨""常规型"三类，三类企业在激励策略上存在本质差异。
3. **Top 排名稳定**：Top 10 企业在两种权重方案下均稳居前列，其中**晶澳科技、领益智造**连续两年方案均为头部。

## 十三、 聚类模型优化与特殊群体深度挖掘

在先前的基线聚类分析中，由于将具有后验综合评价性质的慷慨度得分（Generosity Score）作为输入变量直接参与了聚类，导致多维空间信息发生低效重复与共线性干扰。这使得基线模型划分出的各聚类群体间综合得分差异极小，无法有效抽离出具有明显政策分化和特异化行为特征的样本群体。

本章针对上述局限实施了聚类模型的根本性优化：**切断综合评分的间接干扰，仅提炼 5 个最基础的微观制度参数指标**（授予比例、参与比例、折扣率、正向化等待期、有效期）作为核心输入矩阵，采用 `StandardScaler`（Z-Score 标准化）凸显样本点的多维空间分布方差，并引入 **ANOVA（单因素方差分析）差异检验** 进行统计学健壮性论证。最终，模型成功识别并定量证实了 A 股上市公司股权激励设计的“同质化主流”与“特殊设计群体”。


## 13.1 聚类模型重构与最佳 K 值判定

### 13.1.1 改进微观策略

1. **输入特征纯化**：剔除带有后验与结果性质的 `Generosity Score`，确保聚类矩阵完全基于企业在决策当下的客观制度设计参数，避免“互为因果”带来的算法偏误。
2. **量纲处理优化**：摒弃单纯的 Min-Max 映射，改用 `StandardScaler` 对五个原始指标进行标准化处理，从而真实保留原始数据的分布形态与偏离度。
3. **双准则联合判定**：采用手肘法（Elbow Method，评估误差平方和 SSE 的下降斜率趋势）与轮廓系数（Silhouette Score，评估聚类内部凝聚度与边界分离度）进行联合判定。

### 13.1.2 最佳 K 值选择依据

根据优化后的聚类迭代测试，不同 K 值下的核心评估指标如下表 所示：

|  聚类数 (K)  | 误差平方和 (SSE) | 平均轮廓系数 (Silhouette Score) |
| :-------: | :---------: | :-----------------------: |
| **K = 2** |  **793.73** |       **0.3997（峰值）**      |
|   K = 3   |    649.14   |           0.2431          |
|   K = 4   |    527.85   |           0.2695          |
|   K = 5   |    445.93   |           0.2778          |

**判定结论**：
多维样本空间在 K=2 时表现出最为清晰的结构边界。由表 12-1 可知，**轮廓系数在 K=2 时达到 0.3997 的最高峰值**，说明此时空间分布的紧凑性与隔离度达到了最优平衡；同时，由 K=2 向 K=3 推进时，误差平方和（SSE）的下降斜率边际效应明显减弱。因此，本研究最终锁定 **K=2 **作为最佳聚类数。

## 13.3 聚类结果的 ANOVA 统计显著性差异检验

为了确证由无监督机器学习划分出的两大群体（主流型 vs 特殊型）并非统计学意义上的“随机噪音”或“算法伪像”，本研究对聚类划分结果进行了严谨的单因素方差分析（ANOVA），旨在从统计学上科学地检验两组群间在各维度上的均值差异是否具备显著性。

### 13.3.1 ANOVA 检验输出矩阵

下表详细列示了 5 个聚类输入特征及 1 个后验综合得分在组间的方差检验表现：

| 测试变量类型     | 测试变量指标名称                          | F 统计量 (F Statistic) | P 值 (P Value) |  组间是否具备显著差异 |       统计显著级别      |
| :--------- | :-------------------------------- | :-----------------: | :-----------: | :---------: | :---------------: |
| **聚类输入变量** | **等待期正向化 (`waiting_period_adj`)** |    **1029.1565**    |  **0.000000** | **是（极度显著）** | `***` (p < 0.001) |
| 聚类输入变量     | 授予比例 (`grant_ratio`)              |        3.2561       |    0.072743   |  否（边缘显著趋势）  |       趋近显著边界      |
| 聚类输入变量     | 折扣率 (`discount_rate`)             |        0.8400       |    0.360573   |      否      |       无显著差异       |
| 聚类输入变量     | 参与比例 (`participant_ratio`)        |        0.0894       |    0.765231   |      否      |       无显著差异       |
| 聚类输入变量     | 有效期 (`validity_period`)           |        0.0198       |    0.888234   |      否      |       无显著差异       |

### 13.3.2 检验深度解读

1. **时间锁定期（等待期）的绝对分裂**：
   从表可知，`waiting_period_adj` 的 **F 统计量高达 1029.16，P 值为绝对 0.0000**。这强有力地表明，导致特殊激励设计型群体（Cluster 0）从主流市场方案中完美剥离出来的**核心统计驱动力是时间维度的控制安排**。特殊群体在服务期锁定期、分期解禁的时间节奏上，与主流企业存在实质性的制度鸿沟。
2. **综合分数的非显著性验证与“同分不同质”现象**：
   后验计算的 `Generosity Score` 在组间检验中 P = 0.386，未能通过显著性检验。这一发现恰恰极具公司治理学价值：**它有力地证明了特殊群体并非纯粹的“低慷慨度群体”，而是在综合慷慨度水平与主流群体大致相当的前提下，通过对内部指标的剧烈结构性重组，实现了“同分不同质”的异质化激励架构设计**。
3. **其他维度的合规同质化约束**：
   折扣率、参与比例和有效期的高 P 值，暗示了当前 A 股上市公司在这些维度上受到了强烈的政策法规红线约束（如监管对折价上限、草案有效期的硬性法律规定），从而在多维空间分布上展现出高度集中的趋同行为。


## 13.4 “同质化”与“异质化”的现代公司治理学解释

通过上述对 192 家上市公司股权激励方案的画像重构与统计学检验，本研究揭示了当前 A 股市场深层次的经济含义与公司治理图景：

### 13.4.1 强烈支持“A 股股权激励存在高度同质化现象”

主流均衡激励型（Cluster 1）涵盖了高达 **91.1%** 的样本企业，这在实证层面强烈支持了“当前 A 股上市公司股权激励普遍存在同质化倾向”的假说。

- **治理动机剖析**：这反映出绝大多数科技、制造类上市公司在设计股权激励计划时，存在显著的**制度理论（Institutional Theory）中的模仿同构（Mimetic Isomorphism）行为**。中介机构（券商、管理咨询公司）的方案标准化复制、企业管理层规避监管审查风险的“不求有功但求无过”心态、以及盲目跟从行业头部企业模板的从众心理，共同导致了 90% 以上的草案在设计时直接套用市场成熟模板。

### 13.4.2 少数“特殊激励设计企业”的深度挖掘（Cluster 0）

这 17 家（占比 8.9%）成功突破同质化市场牢笼的企业，展现出了强烈的异质化治理诉求。代表性案例企业包括：**运机集团（001288）**（综合得分 0.504）、**卡莱特（301391）**（综合得分 0.367）和**超讯通信（603322）**（综合得分 0.296）等。它们的治理特征主要体现在：

1. **精细化的非对称授予（“小额短促”策略）**：
   这类企业的 `grant_ratio` 均值（8.53%）仅为主流群体（17.90%）的一半，但其参与比例却能维持在与主流相当的水平。这意味着它们倾向于采用“低总量、高覆盖、快周转”的精密激励手段，而不是依靠盲目做大总量规模来吸引人才。
2. **锁定期机制的柔性定制**：
   ANOVA 检验中极其特殊的 `waiting_period_adj` 表现表明，这 17 家企业并未沿袭主流企业常规的“12个月等待+3年分期解禁”套路，而是选择紧密结合自身科研项目建设周期、核心研发节点的关键里程碑（Milestones），量身定制了更具弹性的服务期锁定或者分批考核解禁机制。

### 13.4.3 经济学总结：从“规模驱动”走向“精准结构设计”

优化后的模型表明：评判一家上市公司的股权激励是否足够高效，不能单纯依赖总体慷慨度分数的绝对值高低。**主流群体通过拉高授予比例等直接手段来获得常规的高评分；而特殊群体虽然授予总规模偏小，但其通过精细化的锁定期设计和精准的激励面覆盖，在内部治理效率上反而可能实现了更高的性价比。** 这种由“规模驱动”向“结构设计”的分化，正是当前高新技术行业在人力资本博弈中走向高级与成熟的典型标志。

## 十四、结论与后续工作

### 14.1 主要结论（聚焦金融实务问题）

本研究聚焦的核心问题是：**A 股上市公司的股权激励方案在"慷慨度"上存在怎样的差异？哪些维度真正决定了慷慨度？投资者应如何解读激励方案的信号？** 基于 200 份股权激励计划草案（清洗后 192 份有效样本）的分析，得出以下实务导向的结论：

#### 结论一：股权激励慷慨度高度分化，头部与尾部差距悬殊

**问题**：上市公司之间的股权激励方案是否"一视同仁"？

**发现**：慷慨度综合得分从 0.033（中远海能）到 0.905（晶澳科技），极差达 **27 倍**。Top 10 企业平均得分 0.58，Bottom 10 平均仅 0.04，**头部企业的激励慷慨度约为尾部的 14 倍**。得分呈明显的右偏分布——70% 的企业得分低于 0.2，仅约 10% 的企业得分超过 0.4。

**实务含义**：投资者不应将"有股权激励"等同于"有激励"。\*\*晶澳科技（0.905）、领益智造（0.744）、柳工（0.582）**等头部企业的激励规模和覆盖范围远超行业平均，而**中远海能（0.047）、安正时尚（0.046）、日辰股份（0.045）\*\*等尾部企业的激励方案实质上更接近"象征性激励"。

#### 结论二：真正决定慷慨度的是"覆盖广度"和"授予规模"，而非定价折扣

**问题**：股权激励的慷慨主要体现在哪些方面？定价折扣、有效期和等待期是否同样重要？

**发现**：熵权法客观赋权显示，**`participant_ratio`（52.75%）和** **`grant_ratio`（40.70%）合计贡献了 93.45% 的信息权重**。而 `discount_rate`（1.48%）、`validity_period`（3.52%）、`waiting_period`（1.55%）三者合计仅 6.55%。这并非说明定价和时间维度不重要，而是说明**在当前样本中，企业间差异最大的恰恰是"激励了多少人"和"每人给多少"这两个实质问题**。

**实务含义**：评估一家公司股权激励的真实价值，应优先关注\*\*"激励对象占员工总数的比例"**和**"授予数量占总股本的比例"\*\*，而非仅看折扣率或有效期。部分企业虽然给出了较高的定价折扣，但如果覆盖范围极窄、授予规模极小，其真实激励价值仍然有限。

### 结论三：A股股权激励呈现“同质化主流”与“异质化特殊”两种对立模式

**1. 主流均衡激励型（同质化群体）**  
占比 **91.1%**（175家），各项指标贴近全样本均值，表现出强烈的制度模仿同构行为，直接套用市场标准模板。

**2. 特殊激励设计型（异质化群体）**  
占比 **8.9%**（17家），授予比例（8.53%）仅为前者一半，等待期（0.4087）不足前者42%，采用“低总量、高覆盖、弹性锁定期”的定制化策略。

**3. “同分不同质”现象**  
两大群体慷慨度得分无显著差异（P=0.386），但等待期差异极端显著（F=1029.16，P≈0）。表明相近的慷慨度可能源于完全不同的结构：主流靠规模驱动，特殊靠精准设计。

**实务含义**：不能盲信慷慨度总分，应优先审视等待期设计是否匹配企业项目周期，识别“真慷慨”与“巧设计”的本质区别。

#### 结论四：慷慨度排名具有稳健性，评估结论可信

**问题**：评分结果是否会因权重选择的不同而大幅波动？

**发现**：熵权法（数据驱动）与人工等权重（主观设定）两种方案的排名 Spearman 相关系数达 **0.777**（p < 0.001）。Top 10 企业在两种权重方案下均稳居前列，Bottom 10 企业同样稳定。

**实务含义**：**本研究的慷慨度排名结论对权重方案不敏感**——无论采用"客观"的熵权法还是"主观"的等权重评估，企业的相对排名基本一致。这意味着本文的 Top 10 / Bottom 10 名单具有较强的鲁棒性，可为投资决策和公司治理评估提供可靠参考。

#### 结论五：数据质量构成实务解读的边界条件

**问题**：评估结果是否存在因数据本身局限导致的偏差？

**发现**：

1. `market_price` 字段存在 2% 的缺失，且部分样本的折扣率出现负值（`discount_rate` 最小 -58.98%），可能意味着**市场价提取存在系统性低估**（公告前一日股价与激励定价基准日不一致）；
2. `participant_ratio` 有 4 条样本经缩尾处理（原始值 > 90%），这些"全员持股"型方案的真实慷慨度可能被略微压低；
3. `waiting_period` 超过 75% 集中在 12 个月（行业惯例），该维度的区分度天然有限。

**实务含义**：在解读排名结果时，需关注**三类潜在偏差**：① 市场价提取偏差可能低估部分企业的定价慷慨度；② 行业覆盖的局限性（仅半导体/电子/计算机/医药/高端制造五个行业）限制了跨行业比较的外推性；③ 样本时间窗口（2021–2025 年）未覆盖完整的牛熊周期。

### 14.2 后续工作建议

1. **人工复核 Top / Bottom 企业**：对 Top 10 与 Bottom 10 企业的原始公告进行人工核验（≥ 20% 抽样），重点验证 `market_price`、`grant_ratio`、`employee_count` 的提取准确性。
2. **补充行业标签**：从 `metadata.csv` 中抽取行业信息（半导体 / 电子 / 计算机 / 医药 / 高端制造），进行**行业间得分差异检验**（Kruskal-Wallis 或分组箱线图）。
3. **时间趋势分析**：按公告年份（2022–2025）分析慷慨度的年度变化趋势，检验是否存在"激励扩容"的政策效应。
4. **LLM Fallback 重抽** **`market_price`**：对 `discount_rate < -10%` 的 7 条样本使用 `--method llm`（Qwen3-8B）重抽 `market_price`，验证是"市场价提取偏差"还是"企业真的高定价"。
5. **可视化增强**：
   - 三维度雷达图（Top vs Bottom vs 中位数企业）
   - 行业分组箱线图
   - 慷慨度年度趋势折线图
   - 三类聚类的特征轮廓图
