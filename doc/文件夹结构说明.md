## 二、逐个文件夹详解

### 📁 `configs/` —— 配置文件夹

| 属性 | 说明 |
|------|------|
| **中文名** | 配置文件夹 |
| **放什么** | `.yaml` / `.json` / `.toml` 等配置文件 |
| **为什么单独放** | 代码和配置分离。改配置不改代码，改代码不改配置。 |

**类比**：就像洗衣机的**控制面板**——你调温度、选模式都在面板上操作，不需要拆开洗衣机的电路板。

**本项目内容**：
- `workflow.yaml`：全局控制面板，定义所有输入/输出路径、API地址、模型参数
- `section_rules.yaml`：段落匹配规则（关键词白名单、黑名单、最小字数限制）

---

### 📁 `src/` —— 源代码文件夹

| 属性 | 说明 |
|------|------|
| **中文名** | 源代码文件夹（src = source 的缩写） |
| **放什么** | `.py` 文件，项目核心逻辑 |
| **为什么叫 src** | 和配置文件分开，方便打包和发布 |

**类比**：就像汽车的**发动机舱**——所有动力都来自这里。

**本项目内容**：

| 文件 | 职责 | 相当于 |
|------|------|--------|
| `common.py` | 读写工具（读YAML/JSONL/CSV，写JSONL/CSV） | 🔧 工具箱 |
| `schemas.py` | Pydantic数据模型定义 | 📋 质检标准 |
| `route_sections.py` | 拼接全文，按规则定位段落 | ✂️  切菜师傅 |
| `extract_fields.py` | 正则提取7个字段 + LLM备用 | 🔍 侦探 |
| `validate_results.py` | 用Schema校验 + 输出CSV | ✅ 质检员 |

---

### 📁 `scripts/` —— 可执行脚本文件夹（已整理）

| 属性 | 说明 |
|------|------|
| **中文名** | 脚本文件夹 |
| **放什么** | 可以直接运行的 `.py` 文件 |
| **为什么单独放** | 与核心逻辑代码（`src/`）区分，这些是"一次性操作"或"流程编排" |

**类比**：就像厨房里的**操作台**——每个脚本是一个做菜步骤。

**本项目内容**：

| 脚本 | 功能 | 什么时候运行 |
|------|------|-------------|
| `test_mineru.py` | 测试 MinerU API 是否通 | 项目最开始，验证API可用 |
| `batch_parse.py` | 批量提交200份PDF给MinerU | 数据获取阶段 |
| `query_batch.py` | 轮询检查MinerU处理进度 | 提交后等待结果 |
| `download_md.py` | 下载解析好的MD文件并解压 | 处理完成后 |
| `build_parsed_docs.py` | 将MD文件+metadata组装成一个大文件 | MinerU解析完成后 |
| `collect_md.py` | 收集整理Markdown文件 | 需要整理MD文件时 |

---

### 📁 `data/` —— 数据文件夹

| 属性 | 说明 |
|------|------|
| **中文名** | 数据文件夹 |
| **放什么** | CSV表格、JSONL数据、数据库文件等**不可执行**的数据 |
| **核心原则** | data 文件夹里只放数据，不放代码！ |

**类比**：就像厨房的**冰箱**——里面只有食材（数据），没有厨具（代码）。

**本项目内容**：

```
data/
├── metadata/          ← 元数据（"数据的数据"，描述数据来源和信息）
│   └── metadata.csv   ← 200条公告的基本信息表
│
├── parsed/            ← 解析过程中的中间产物
│   ├── parsed_docs.jsonl   ← 把MD文件和元数据合并后的结果
│   └── sections.jsonl      ← 全文拼接后的结果
│
└── batch/             ← MinerU批量处理的追踪记录（仅CSV数据文件）
    ├── batch_mapping.csv   ← 哪个公告对应哪个MinerU任务
    ├── result_mapping.csv  ← 每个任务的处理状态（done/processing/failed）
    └── state.csv           ← 状态汇总
```

> ⚠️ **重要约定**：`data/` 里的文件**不应该被 Git 忽略**（尤其是 metadata.csv 和中间产物），但 `downloads/` 和 `markdown/` 可以忽略（因为可以重新生成）。

---

### 📁 `markdown/` —— MinerU 解析产出文件夹

| 属性 | 说明 |
|------|------|
| **中文名** | Markdown 文件夹 |
| **放什么** | MinerU 从 PDF 解析出来的 `.md` 文件 |
| **文件命名** | `<doc_id>.md`（如 `1224831097.md`） |

**为什么叫 markdown？** 因为 MinerU 把 PDF 转成了 Markdown 格式（一种轻量级标记语言，比纯文字多了标题、表格等格式）。

```markdown
...
## 二、股票来源
...
## 三、授予数量
拟授予股票期权 3,186.2261 万份，约占本激励计划公告时公司总股本的 3.50%。
...
```

### 📁 `downloads/` —— MinerU 原始下载文件夹

| 属性 | 说明 |
|------|------|
| **中文名** | 下载文件夹 |
| **放什么** | MinerU 返回的原始 ZIP 包 + 解压后的内容 |
| **特点** | 这是**原始数据**，可以随时重新下载 |

**结构示例**：
```
downloads/
├── 1214825162.zip          ← MinerU返回的ZIP包
├── 1214825162/             ← 解压后的目录
│   ├── markdown.md         ← 解析出的Markdown文件
│   ├── model.json          ← MinerU的模型元数据
│   ├── layout.json         ← 页面布局信息
│   ├── content_list.json   ← 内容列表
│   └── images/             ← 从PDF中提取的图片
│       ├── 015cdcbc...jpg
│       └── ...
└── ...
```

> 这个文件夹通常**不需要**提交到 Git（可以加到 `.gitignore`），因为里面的内容可以从 MinerU 重新下载。

### 📁 `outputs/` —— 最终产出文件夹

| 属性 | 说明 |
|------|------|
| **中文名** | 输出文件夹 |
| **放什么** | 项目的**最终成果**和**运行日志** |
| **子文件夹** | `results/`（结果）、`reports/`（报告）、`logs/`（日志） |

**类比**：就像餐厅的**出餐口**——做好的菜从这里端出去给客人。

```
outputs/
├── results/                    ← 🎯 核心产出
│   ├── extract_results.jsonl   ← 提取的原始结果（含证据文本）
│   ├── records_validated.csv   ← ✅ 校验通过的198条记录（可拿去做分析）
│   └── all_200_records.csv     ← 全部200条记录（含提取失败的）
│
├── reports/                    ← 📊 检查报告
│   └── section_check_report.csv ← 段落路由检查结果
│
└── logs/                       ← 📜 运行日志
    └── validation_errors.jsonl  ← 校验失败的记录及原因
```

### 📁 `venv/` —— Python 虚拟环境文件夹

| 属性 | 说明 |
|------|------|
| **中文名** | 虚拟环境文件夹（venv = virtual environment） |
| **放什么** | Python 解释器 + 项目依赖的第三方库 |
| **是否需要手动改** | ❌ 不需要！由 `pip install` 自动管理 |

**类比**：就像每个餐厅有**自己独立的厨房用具**——不会和隔壁餐厅的混在一起。

**为什么需要它？** 因为不同项目可能需要不同版本的 Python 库。venv 让每个项目有自己独立的 Python 环境。

> ⚠️ **绝对不要**提交到 Git！这是 Python 项目的第一条军规。

## 三、常见项目文件夹命名规范

在 GitHub 上看到其他 Python 项目时，你会经常遇到这些文件夹名：

| 文件夹名 | 全称 | 中文 | 放什么 |
|----------|------|------|--------|
| `src/` | source | 源码 | 项目的核心 Python 代码 |
| `lib/` | library | 库 | 自己写的工具库 |
| `config/` 或 `configs/` | configuration | 配置 | YAML/JSON/TOML 配置文件 |
| `data/` | data | 数据 | CSV/JSONL/数据库等数据文件 |
| `output/` 或 `outputs/` | output | 输出 | 运行结果、报告、日志 |
| `logs/` | logs | 日志 | 程序运行日志 |
| `docs/` | documentation | 文档 | 项目文档（不只代码注释） |
| `tests/` | tests | 测试 | 单元测试代码 |
| `scripts/` | scripts | 脚本 | 一次性运行的工具脚本 |
| `notebooks/` | notebooks | 笔记本 | Jupyter Notebook（.ipynb 文件） |
| `assets/` | assets | 静态资源 | 图片、字体等静态文件 |
| `venv/` 或 `.venv/` | virtual environment | 虚拟环境 | Python 虚拟环境 |
| `dist/` | distribution | 分发 | 打包后的安装包 |
| `build/` | build | 构建 | 编译/构建产物 |

## 四、本项目整理的改动说明

为了让你更容易理解项目结构，我对文件夹做了以下整理：

| 改动 | 原因 |
|------|------|
| ✅ 新建 `scripts/` 文件夹 | 把所有可执行脚本集中在一起（原来散落在根目录和 `data/batch/`） |
| ✅ 删除 `Untitled-1.py` | 空文件，没有用 |
| ✅ 删除 `output_md/` | 空文件夹 |
| ✅ 删除 `mineru_md_output/` | 空文件夹 |

| ✅ 从 `markdown/` 移除 `md.zip` | 压缩包不属于这里 |

## 五、快速定位指南

> 当你想……的时候，去哪个文件夹找？

| 你想做什么 | 去哪里 |
|-----------|--------|
| 改 API 秘钥 | `.env` |
| 改输入输出路径 | `configs/workflow.yaml` |
| 改字段提取规则 | `src/extract_fields.py` |
| 改数据模板 | `src/schemas.py` |
| 看原始数据 | `data/metadata/metadata.csv` |
| 看最终结果 | `outputs/results/records_validated.csv` |
| 看谁提取失败了 | `outputs/logs/validation_errors.jsonl` |
| 看 MinerU 解析出的原文 | `markdown/md/<doc_id>.md` |
| 看 MinerU 原始返回 | `downloads/<doc_id>/` |
| 重新运行提取 | `python src/extract_fields.py --config configs/workflow.yaml` |
| 重新运行全流程 | 按方法链详解里的8步命令依次执行 |
